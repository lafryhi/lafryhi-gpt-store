# Product Standard

This document defines the unified standard for all GPT products inside **LAFRYHI GPT Store**.

## Product Identity

Each product has four independent identities:

- `Product ID`
- `Internal Slug`
- `OpenAI GPT ID`
- `Public URL`

### Identity rules

- `Product ID` is the internal catalog identifier, for example `LAF-001`.
- `Internal Slug` is the internal project reference used across product files and store metadata.
- `OpenAI GPT ID` is the external stable identifier assigned by OpenAI.
- `Public URL` is the user-facing access link for the published GPT.
- Do not assume that the text portion of the ChatGPT URL must match the internal slug.
- The OpenAI GPT ID is the authoritative external reference for the GPT resource.

## Product Lifecycle

Every product moves through these stages in order:

1. `Idea`
2. `Specification`
3. `Prompt`
4. `Knowledge`
5. `Testing`
6. `Publishing`
7. `GPT Store`
8. `Maintenance`

### Stage definitions

- `Idea`: Define the core problem and product goal.
- `Specification`: Write the description, use cases, and limits.
- `Prompt`: Prepare the complete System Prompt.
- `Knowledge`: Add reference material if needed.
- `Testing`: Validate behavior with practical test cases.
- `Publishing`: Prepare the product for publishing inside ChatGPT.
- `GPT Store`: Add the product data to the store catalog.
- `Maintenance`: Review and update the product over time.

## Required fields for each GPT

Every GPT in the system must include:

- `Product ID`
- `Product Name`
- `Internal Slug`
- `OpenAI GPT ID`
- `Public URL`
- `Category`
- `Version`
- `Status`
- `Owner`
- `Created Date`
- `Last Updated`

## Field rules

### Product ID

- Use the unified format: `LAF-XXX`
- The number must be three digits
- Example: `LAF-001`

### Product Name

- Use a clear and direct product name
- The name should match the product identity used in store materials

### Internal Slug

- Use English only
- Use lowercase letters only
- Use hyphens instead of spaces
- Do not use special symbols
- Keep it short and readable
- It must be unique inside the project

### OpenAI GPT ID

- Use the stable GPT identifier generated by OpenAI
- Example format: `g-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`
- Do not invent or modify this value

### Public URL

- Use the live ChatGPT URL shown after publishing
- This is the user-facing link for the GPT
- Keep it in sync with the published GPT resource

### Category

- Use one primary category per product
- The category should be stable and consistent across the store

### Version

- Follow the release policy defined in `RELEASE_POLICY.md`

### Status

- `Idea`
- `Development`
- `Testing`
- `Ready`
- `Published`
- `Maintenance`
- `Coming Soon`
- `Active`

### Owner

- The person or team responsible for the product

### Created Date

- Use a clear and stable date format
- Example: `2026-07-07`

### Last Updated

- Use the date of the most recent update or review

## Slug naming rules

1. Use English only.
2. Use lowercase letters.
3. Replace spaces with hyphens.
4. Avoid unclear abbreviations.
5. Keep the slug as short as possible while preserving clarity.
6. Do not reuse an existing slug.

### Correct examples

- `gpt-sotour`
- `ai-radar`
- `codex-assistant`

### Incorrect examples

- `GPT SOTOUR`
- `GPT_Sotour`
- `gpt sotour`

## Product numbering rules

1. Start the sequence at `LAF-001`.
2. Each new product gets the next number in order.
3. Do not reuse deleted numbers.
4. Do not renumber products after assignment unless there is a strong organizational reason.
5. Keep the number stable in all reference files.

## Product readiness rule

A product is not considered ready for the store unless:

- The specification is complete
- The prompt is complete
- Test cases are prepared
- Quality review is complete
- The owner is clear
- The final version is approved
